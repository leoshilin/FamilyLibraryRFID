# 命名规范审查报告（DB 设计文档 + API 接口文档）

> 审查日期：2025-07-11
> 审查范围：`Docs/家庭图书管理系统数据库表结构设计.md`、`Docs/api-service-接口说明.md`
> 对照规范：
> - **API 入参 / 返回（前端实体）：全部 camelCase**

## 一、结论速览

- **数据库设计文档**：存在 **4 处**字段用了 camelCase（应为 snake_case）。
- **API 接口文档**：存在 **多处**契约字段用了 snake_case（应为 camelCase），分布在 **返回** 与 **入参** 中，且部分在**代码侧同样违反**（不是纯文档错误）。
- 修复1 已统一的 `rfid_tid` / `rfidTid` 字段，本次确认无违规残留。
- 文档 0.2 / 遗留问题 2 已承认 `item_id`/`family_id` 不一致，但本次审查发现**实际违反范围更广**，建议扩大统一范围（见第四节）。

## 四、与文档既有声明的对照 & 建议

1. **文档 0.2 / 遗留问题 2** 已承认 `itemId`↔`item_id`、`familyId`↔`family_id` 不一致，并标注"下一阶段统一"。本次审查确认该不一致**真实存在且范围更大**，建议统一范围扩至：`itemId`/`item_id`、`familyId`/`family_id`、`bookshelfId`/`bookshelf_id`、`bookMetaId`/`book_meta_id`、`coverUrl`/`cover_url`、`sortOrder`/`sort_order`、`publishYear`/`publish_year`、`isSet`/`is_set`、`setTotalCount`/`set_total_count`、`bookshelfName`/`bookshelf_name`，以及时间类 `createdAt`/`created_at`、`updatedAt`/`updated_at`、`createdBy`/`created_by`。

2. **DB 设计文档是根因之一**：`user.currentFamilyId`、`user_family.userId`/`familyId`、`bookshelf.familyId` 在 DB 层就是 camelCase，导致 API 返回与代码自然沿用。需先改 DB 设计文档 → 改代码 DB 写入字段 → 改 API 返回映射（snake→camel）。

3. **`api_user_login` 直接返回原始 `user` 文档**：会透出 DB 原始命名（含 `updated_at`/`created_at`/`currentFamilyId`）。建议增加 `toUserEntity` 映射器统一转 camelCase，与 `toBookItemEntity`/`toBookMetaEntity` 保持一致。

4. **修复1 已统一项**：`book_item.rfid_tid`（DB）/ API 返回 `rfidTid`（E2/E6/F1/F2）/ 前端 `rfidTid` 已全链路一致，本次确认无违规残留。

## 五、建议的处理顺序（如需修复）

1. 先改 **DB 设计文档** 的 4 处 camelCase 字段 → 同步改代码 DB 写入（`api_user_*`、`api_user_family_*`、`api_bookshelf_*`）。
2. 给 `user` 增加 `toUserEntity`，消除 `api_user_login` 透传原始文档的问题。
3. 统一 **API 返回** 映射：补齐 `toFamilyEntity`/`toBookshelfEntity`/`toBookMetaEntity` 的 camelCase 化（`createdBy`/`createdAt`/`updatedAt`/`sortOrder`/`coverUrl` 等）。
4. 统一 **API 入参**（`itemId`、`coverUrl` 等），并改对应代码读取。
5. 同步更新 `api-service-接口说明.md` 全部返回/入参示例，并删除 0.2 中"RFID 已统一、其余待统一"的过时描述。

> 本报告仅列示违反项，未做任何代码/文档修改。

## 六、处理进展（路线 B 已实施，2025-07-11）

经评估，采取**路线 B（映射器方案，不动 DB 字段）**而非路线 A（先改 DB 再改 API）：

- **不动 DB 设计文档与代码 DB 写入**：`user.currentFamilyId`、`user_family.userId`/`familyId`、`bookshelf.familyId` 这 4 个 DB 内部 camelCase 字段保留，作为**独立迁移专项**后续处理（见 API 文档 0.2）。
- **已完成**：
  1. API 入参 / 返回（前端实体）全链路统一为 camelCase：`itemId`/`familyId`/`bookshelfId`/`bookMetaId`/`coverUrl`/`sortOrder`/`createdAt`/`updatedAt`/`createdBy`/`bookshelfName`/`publishYear`/`isSet`/`setTotalCount` 等。
  2. 新增 `toUserEntity` 映射器，消除 `api_user_login` 透传原始 DB 文档的问题（与 `toBookItemEntity`/`toBookMetaEntity`/`toBookshelfEntity` 一致）。
  3. 前端 `miniprogram` 实体发送与读取全部对齐 camelCase，无 snake_case 残留。
  4. `api_bookitem_create` 的 `normalizeInput` 入参读 camelCase、出 DB snake_case；调用点（`book` 对象）相应改为 camelCase 键。
  5. `api-service-接口说明.md` 全部请求/返回示例已改为 camelCase；0.2/0.3 与"遗留问题"第 2 条已更新为"命名统一专项（已完成）"，并标注 DB 内部 4 个 camelCase 字段为独立迁移专项。
- **未做（路线 B 范围外）**：DB 设计文档 4 处 camelCase 字段重命名为 snake_case（路线 A），需独立排期 + 迁移脚本 + 全量回归（影响 ~30 个云函数对 `user.currentFamilyId` 等的依赖）。

> 边界原则：snake_case 的 DB 与 camelCase 的 API 之间一律由映射器（`toUserEntity`/`toBookItemEntity`/`toBookMetaEntity`/`toBookshelfEntity`/`normalizeInput`）转换，DB 字段重命名不阻塞 API/app 一致性。
