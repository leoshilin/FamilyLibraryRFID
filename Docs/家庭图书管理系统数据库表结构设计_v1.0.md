# 家庭图书管理系统 – 数据库表结构设计文档 V1.0

> 2026/1/15 发布

## 一、整体设计原则

1. 家庭（Family）是系统的第一业务边界

2. 系统级主数据（book_meta）与家庭级实体（book_item）严格分离

   * 系统级：书籍在“世界层面”的客观定义（跨家庭共享），定义书籍“是什么”（book_meta）

   * 家庭级：家庭“拥有什么书”（book_item），家庭中真实存在的一本实体书

3. 套装书是系统级出版形态，不是库存属性 （=套装书是系统级概念，不允许家庭私有定义）

4. RFID 为可选能力，允许为空、可重绑定

5. 所有关键操作必须可审计（时间 + 操作人）

6. 所有影响 book_meta 的变更必须可审批、可回滚

## 二、字段类型统一规范

| 类型         | 说明                      |
| ---------- | ----------------------- |
| varchar(n) | 有长度约束的字符串（ID / 名称 / 编码） |
| text       | 长文本                     |
| json       | 结构化可扩展字段                |
| enum       | 枚举值                     |
| boolean    | 布尔                      |
| int        | 整数                      |
| datetime   | UTC 时间                  |

## 三、表结构设计（修订后）

### 1️⃣ user（用户表）

| 字段名        | 类型           | 中文说明                  |
| ---------- | ------------ | --------------------- |
| id         | varchar(64)  | 用户主键 ID（系统生成）         |
| openid     | varchar(64)  | 微信 openid（应用内唯一）      |
| unionid    | varchar(64)  | 微信 unionid（跨应用唯一，可为空） |
| nickname   | varchar(100) | 用户昵称                  |
| created_at | datetime     | 创建时间                  |

**唯一性规则说明：**

* openid 在同一小程序内唯一

* unionid 若存在则用于跨系统用户合并

* 系统内部统一使用 id 作为外键

**设计预留说明：**

* 本系统当前版本直接使用 openid 作为登录身份标识。

* 若未来支持多登录来源，可引入 user_auth 表，将 openid / unionid 迁移至该表。

## 四、关键索引建议

* book_meta.isbn

* book_meta.is_set

* book_item.family_id + status

* book_item.rfid_tag_id

* family_member.family_id + user_id

* operation_log.family_id + created_at


## 五、关键修订总结

1. 统一字段类型为 varchar / enum / json 等，避免 string 歧义

2. user 表采用 **id + openid + unionid** 三层唯一性设计
