## E-8【P0 严重·跨文档冲突】`inventory_status` vs `status` 字段命名不一致

这是本次核对发现的**最高优先级问题**，横跨数据库设计文档、API 文档与代码三方。

- **数据库设计文档**（`家庭图书管理系统数据库表结构设计.md` 3.6，L182）定义 `book_item` 字段为：
  `inventory_status | enum | 必须 | in_stock / off_stock`
- **API 文档 A17**（P723-725）沿用：`inventory_status = in_stock`
- **代码侧却分裂为两套**：
  - 使用 `status` 的函数：`api_bookitem_create`（写入 `status:'in_stock'`, L101）、`offstock`、`restock`、`delete`、`api_recentbook_search`（L2187）、`api_book_search`（L940）
  - 使用 `inventory_status` 的函数：仅 `api_bookshelf_delete`（L1317）、`api_bookshelf_list`（L1447）

- **可能后果（取决于线上真实字段名）**：
  - 若真实字段是 `status`：则 `bookshelf_delete`/`bookshelf_list` 的"在架图书检查"永远查不到（`inventory_status` 无值）→ **有书的书架也能被删除**（严重数据/权限 bug）；且数据库设计文档、API 文档 A17 均错。
  - 若真实字段是 `inventory_status`：则 `create/offstock/restock/delete/recent/search` 全部读写错字段，首页"最近上架"永远为空——这种程度的问题几乎不可能通过测试，概率极低。
- **建议（紧急）**：以线上数据库实际 schema 为唯一真相，统一为单一字段名，并**同步修正**三处：①数据库设计文档 3.6；②API 文档 A17；③`api_bookshelf_delete` / `api_bookshelf_list` 代码。在统一前，A17 的"检查在架图书"逻辑不可信。

- **Todo: 修改代码统一字段: done**

# 3. 从架构层面的调整建议

你现在实际上已经出现一个现象：

prepareInStock
commitInStock

两个函数共同完成一次上架

而：

offBookItem
onBookItem
deleteBookItem

又是三个独立函数

从DDD（领域设计）角度看其实不完全对称。

我会建议最终形成：

api.bookitem.create
api.bookitem.updateStatus
api_bookitem_delete
api_bookitem_get

api_book_search
api.book.recent

api_bookmeta_getByIsbn
api_bookmeta_fetchExternal

其中：

updateStatus

统一处理：

{
  "action":"off_stock"
}
{
  "action":"restock"
}

以后：

{
  "action":"bind_rfid"
}
{
  "action":"lost"
}

都能扩展。

不过这是第二阶段优化。

对于你现在的开发进度，我建议：

先不要合并。

保持：

offBookItem
onBookItem
deleteBookItem

三条独立API。

这样逻辑最清晰，也最容易调试和排查问题。

等 RFID、Task、Inventory 全部完成后，再考虑 API 收敛与统一。这样风险最低。
